LMNT Framework

A reusable full-stack platform for building business applications without rebuilding authentication, permissions, tenancy, navigation, communication, workflows, observability, and the application shell for every product.

Foundation

How LMNT is structured

LMNT is delivered as two reusable packages. A product application depends on them and adds only its own business domain.

Backend framework

com.titan:lmnt

Spring Boot auto-configuration, security, persistence, controllers, services, realtime messaging, workflow support, audit and platform integrations.

Frontend framework

@titan3rd/framework-ui

React application shell, login, dynamic navigation, protected routing, platform pages, notifications, chat, direct calls, theming, localization and reusable exports.

LMNT backendCommon platform APIs and services.
Product backendBusiness entities, controllers and rules.
LMNT UI shellAuthentication, layout and platform screens.
Product pagesProduct-specific center content and routes.
Framework boundary
If multiple products can reuse a capability, it may belong in LMNT. If only one product needs it, keep it in the product repository.
User experience

Typical user journey

1

Sign in

The login screen loads dynamic product branding and creates a secure cookie-based session.

2

Resolve context

LMNT loads the current user, tenant, active organization, memberships and effective permissions.

3

Build navigation

The backend returns only framework and product menus available in the current permission and subscription context.

4

Connect realtime

The shell opens a SockJS/STOMP connection for presence, notifications, chat and call signalling.

Authentication

Login and session behavior

The login page is framework-owned but product-aware.

Dynamic login content

  • Application name, version, channel and product description are loaded from /api/app-info.
  • The product logo uses productLogoPath, then /lmnt-product-logo.svg, and finally the built-in LMNT logo if the image fails.
  • A product description replaces the default login subtitle when one is registered.
  • The footer shows “Powered by Framework LMNT”, the active application version and the current year.

Authentication sequence

CredentialsUsername and password are submitted to /api/auth/login.
Optional locationThe browser may supply latitude/longitude. Denial does not block login.
Secure cookieThe server sets an HttpOnly JWT cookie with SameSite=Lax.
WorkspaceThe theme is reapplied and navigation continues to the product default route.

Session properties

  • The default cookie name is LMNT_TOKEN.
  • The cookie is HttpOnly, scoped to /, and uses SameSite=Lax.
  • LMNT_AUTH_COOKIE_SECURE=true is required for HTTPS-only production cookies.
  • The token is refreshed during authenticated requests, producing a sliding session window.
  • The returned LMNT activity session ID is stored locally only to finalize presence and activity during logout.
  • API clients may use a Bearer token because the backend falls back to the Authorization header when no cookie exists.
Current source limitation
The “Forgot your password?” control is present in the form but links to #. A password-reset workflow must be supplied before presenting it as functional.

Expired sessions

Protected routes call /api/me. When the user is missing or an API request returns an authentication failure, the shell redirects to /login and displays a “Session expired” notification.

Application shell

Header and dynamic branding

The header remains visible across authenticated platform and product routes.

Left side

  • Sidebar collapse/expand control.
  • A one-time welcome animation per browser tab session.
  • Permanent active tenant and organization labels after the welcome animation.

Right side

  • Dynamic application name and version.
  • Build channel badge such as dev.
  • Unread notification counter and quick actions.
  • Help and About links.

Help and About dynamicity

Products can replace both links through createLmntApp({ headerLinks }). Without overrides, Help opens this manual and About opens the Titan3rd GitHub profile.

Navigation

Sidebar and menu dynamicity

The sidebar is not a hardcoded list. It is assembled from the backend for the current user context.

How the menu is built

  1. The frontend requests GET /api/menu.
  2. The backend loads framework menu rows from lmnt_app_menu.
  3. When LMNT_PRODUCT_CODE is configured, it also loads the active product’s lmnt_product_menu rows.
  4. Rows are filtered by active permission, current product, tenant, organization and product-subscription access.
  5. Empty parent sections are removed unless at least one child remains visible.
  6. The frontend sorts entries by displayOrder, translates labels, resolves Lucide icon names and renders collapsible groups.

Visible behavior

  • The entire sidebar can collapse into icon-only mode.
  • Section headings can expand and collapse while the sidebar is open.
  • The search field filters both direct items and nested children.
  • Active routes are highlighted.
  • Product branding at the top uses the same dynamic name and logo as login.
  • The footer contains the active user and organization selector.
A route requires both registration and menu authorization
Adding a React route does not make it reachable. The same URL must be returned by /api/menu, normally through a framework or product menu row connected to a permission granted in the current context.

Menu Management screen

The Menu Management page can create, edit and delete:

  • Heading sections — non-clickable groups that contain items.
  • Menu items — clickable routes with title, URL, icon, order and permission.
  • Framework menus — reusable platform navigation.
  • Product menus — navigation owned by a registered LMNT product.

After a successful change, the UI dispatches an internal menu-refresh event so the sidebar can reload without a full sign-out.

Identity context

User menu and organization switching

The user control at the bottom of the sidebar is both an account menu and the active organization selector.

Organization selector

Lists active organization memberships, organization logos and roles assigned in each membership.

Switching calls /api/v1/me/switch-org, replaces the active organization claim, reloads the user context and refreshes the workspace.

Account controls

Displays profile details, encrypted user ID, contact data, tenant, addresses, timezone, manager, roles and direct permissions.

Also links to notifications, conditionally exposes Settings and performs logout.

Password change

A user may change their password from the account dialog. The new password must:

  • Contain at least eight characters.
  • Contain uppercase and lowercase letters.
  • Contain a number and a special character.
  • Differ from the current password.

After a successful change, LMNT signs the user out and requires a new login.

Context changes permissions
A user may see different routes and actions after changing organizations because organization-membership roles replace global user roles when an active organization context exists.
Access control

Permissions and route authorization

LMNT evaluates permissions on both the backend and frontend. UI hiding is not treated as security enforcement.

Permission key model

A permission is represented as RESOURCE_ACTION, for example:

Permission examples
USER_MANAGEMENT_VIEW
USER_MANAGEMENT_ADD
MENU_MANAGEMENT_EDIT
SUPPORT_TICKETS_MANAGE
HMS_BOOKINGS_MENU_MANAGE

Where permissions come from

  • Direct permissions assigned to the user.
  • Permissions inherited from active roles.
  • When an organization is active, roles assigned through the user’s organization membership.
  • Super Admin receives every active permission.

Scope matching

Scope dimensionBehavior
ProductA permission with no product is reusable globally; a product-bound permission applies only to that product runtime.
TenantA null tenant is global; a tenant-bound permission must match the authenticated user’s tenant.
OrganizationA null organization is global; an organization-bound permission must match the active organization.

Enforcement layers

  • Backend: @RequiresPermission("RESOURCE_ACTION") rejects unauthorized API calls.
  • Menu: unauthorized menu items are not returned by /api/menu.
  • Route shell: authenticated URLs missing from the allowed menu paths show an Access Denied page.
  • UI actions: usePermission() and <Can> hide or disable buttons for unavailable actions.

Modules

Platform and observability

Home

The framework Home page is an interactive quick-start reference covering overview, packages, local services, environment variables, product structure and security rules.

Dashboard

The security dashboard combines:

  • Current tenant, organization and encrypted identity metadata.
  • Direct and effective permission clearance.
  • Realtime active-user presence and idle duration.
  • Separation-of-duties and identity-scope discrepancy indicators.
  • API metrics, audit events and recent message-queue traces.

Analytics

Provides system CPU, process CPU, JVM heap/non-heap memory, Redis memory, threads, uptime, request history and connectivity summaries for MySQL, Redis, MinIO, Ollama, RabbitMQ, LiveKit and Razorpay.

API Trace

Displays RabbitMQ exchange, routing key, queue, status, duration and errors. It depends on MQ trace interception and stored trace records.

Modules

Identity and administration

Users

Create and update profiles, crop profile images, assign a primary tenant/organization, add multiple organization memberships, assign roles and direct permissions, enable/disable accounts and inspect activity/API summaries.

Roles

Create global, tenant or organization-scoped roles and manage assigned permissions through paginated search panels.

Permissions

Define resource, action, scope, product, tenant and organization. Permissions can be active/inactive and may be marked visible or grantable to tenants.

Tenants & organizations

Tenants form isolation boundaries. Organizations live within tenants and supply the active operational context used by memberships and permission evaluation.

Modules

Workflow, Board and calendar

Workflow engine

LMNT integrates Flowable using a separate configurable datasource. Product features can bind to process-definition keys without duplicating BPMN XML in product tables.

  • Definitions: deploy, list, inspect XML, suspend and activate process definitions.
  • Bindings: connect a product feature key to a process definition.
  • My Tasks: assigned and claimable tasks with completion variables.
  • Integration API: start-or-get by business key, correlate messages, update variables, complete tasks and cancel instances.
  • Audit: registry state plus detailed historic activities.

The Board

The Board implements projects and Kanban tasks with Backlog, To Do, In Progress, Review and Done transitions. Tasks can carry type, priority, assignee, business key and workflow instance metadata.

Calendar providers

The calendar service aggregates registered CalendarEventProvider implementations. Framework providers currently include Board task deadlines and team-availability events, while products may add more providers.

Modules

Communication

Chat

  • Direct conversations and group conversations.
  • Create, rename and delete groups subject to authorization.
  • Unread/read state for direct and group messages.
  • Presence, last-seen information and typing indicators.
  • Reply context and rich-text shortcuts.
  • Attachments stored through the framework attachment service.
  • Audio/video call controls when direct-call infrastructure is available.

AI Chat

Streams responses as Server-Sent Events from /api/ai/chat, checks AI health, lists Ollama models, permits model selection and supports stopping a response. Availability depends on the configured AI server.

The Wall

An internal community feed for announcements and events. It supports cover images, date ranges, locations, pinning, reactions, comments, RSVP and notification-linked deep navigation.

Modules

Finance and support

Payments

Razorpay-backed payment history and order views include search/filtering, CSV/Excel/PDF export, checkout creation, signature verification and QR-code generation.

Product Subscriptions

Subscriptions can restrict product menus and APIs using scope, status, dates, allowed permission keys and allowed URL prefixes. Current product access is evaluated before product menus are returned.

Tickets

Users can create and follow support tickets. Support managers can route by user, product or role, change status, assign ownership, complete tickets, attach files and create linked Board tasks.

Preview modules
Payroll and Expense Claims are intentionally presented as demo/reference UI in the current source. Do not describe their dummy data, uploads or payout actions as production functionality.

Realtime systems

Realtime architecture

Authenticated pages share one realtime provider so notifications, chat, presence and direct-call signalling use the same connection.

Connection behavior

  • SockJS endpoint: /chat-websocket.
  • STOMP reconnect delay: approximately 3 seconds.
  • Incoming and outgoing heartbeats: 20 seconds.
  • Presence refresh: every 30 seconds while connected.
  • Visible/focused tabs reconcile notifications and presence after returning to the foreground.
  • Optional Redis realtime bridge supports events across multiple application instances.
Destination/actionPurpose
/user/queue/notificationsPersonal notification delivery.
Resolved broadcast topicsTenant/product/organization broadcast invalidation followed by notification reconciliation.
/user/queue/callsIncoming, ringing, accepted and terminal call events.
/app/presence.refreshRefreshes active-user presence.
/app/chat.viewReports whether the visible page is the active chat view.

Communication scope

lmnt.scope.chat and lmnt.scope.calls accept:

ValueReach
GLOBALShared communication scope without product, tenant or organization restriction.
PRODUCTCurrent product, tenant and organization.
ORGCurrent tenant and organization across products through shared identity records.
TENANTCurrent tenant across products through shared identity records.

The current property defaults are GLOBAL. Products should explicitly choose the required isolation level rather than relying on defaults.

Realtime

Notifications

LMNT supports personal and broadcast notifications with title, message, severity/type, subject, deep link, read state and creation time.

Header notification control

  • Unread count badge.
  • Latest unread items with severity color.
  • Relative timestamps.
  • Open notification link and mark that notification read.
  • Mark one or all notifications read.
  • Open the full Notifications page.

Full Notifications page

Provides All/Unread and severity filters, refresh, bulk read action and deep-link navigation. The client reconciles with /api/notifications/all/me after connection, focus and visibility changes.

Realtime

Audio and video calls

STOMP carries call signalling; LiveKit carries microphone, camera and media-room traffic.

StartCaller requests audio or video call for a username.
SignalCallee receives an incoming event on the calls queue.
AcceptBoth sides obtain a short-lived LiveKit token after acceptance.
MediaParticipants join the generated LiveKit room until end/reject/cancel.

Call states

idle, connecting, incoming-ringing, outgoing-ringing, in-call, ended and error.

Terminal backend states include rejected, cancelled, missed, ended and failed. Accepted calls retry token retrieval briefly to handle signalling/order timing.

LiveKit configuration

  • LMNT_LIVEKIT_URL
  • LMNT_LIVEKIT_API_KEY
  • LMNT_LIVEKIT_API_SECRET
  • LMNT_LIVEKIT_CALL_TIMEOUT_SECONDS
Browser behavior

Current realtime limits

Browser stateCurrent expected behavior
Tab visible and activeRealtime notifications, presence, chat and calls operate through the active socket.
Tab open but unfocused/hiddenThe socket normally remains connected, but browser throttling and device power policies can delay timers or networking.
Browser minimizedUsually similar to a hidden tab while the browser process remains alive.
Tab or browser closedNo socket is active, therefore incoming calls and in-app notifications cannot be delivered by the current frontend.
Web Push is not implemented in this source snapshot
The ZIP defines lmnt.web-push.public-key, private-key and subject properties, but it contains no browser service worker, PushSubscription registration UI or backend push-dispatch implementation. These properties are reserved configuration only in version 26.07.12.
Development

Building a product on LMNT

Frontend entry point

ui/src/main.tsx
import { StrictMode } from "react"
import { createRoot } from "react-dom/client"
import { createLmntApp } from "@titan3rd/framework-ui"
import "@titan3rd/framework-ui/styles.css"
import packageJson from "../package.json"
import { productRoutes } from "./routes"

const App = createLmntApp({
  productRoutes,
  defaultRoute: "/product/dashboard",
  productLogoPath: "/lmnt-product-logo.svg",
  productFrontendVersion: packageJson.version,
  headerLinks: {
    helpUrl: "https://titan3rd.github.io/mywebsite/",
    aboutUrl: "https://example.com"
  }
})

createRoot(document.getElementById("root")).render(
  <StrictMode><App /></StrictMode>
)

Route registration

ui/src/routes.tsx
import type { LmntProductRoute } from "@titan3rd/framework-ui"
import { ProductDashboard } from "./pages/ProductDashboard"

export const productRoutes: LmntProductRoute[] = [
  { path: "/product/dashboard", element: <ProductDashboard /> },
  { path: "/product/public", element: <PublicPage />, requiresAuth: false }
]

Four pieces required for a protected product page

  1. Register the React route in productRoutes.
  2. Register the product in lmnt_product and set LMNT_PRODUCT_CODE.
  3. Create a matching lmnt_product_menu URL.
  4. Attach and grant the corresponding permission in the required product/tenant/org context.

Product metadata

The backend /api/app-info response uses the configured product record for name, code and description, while resolving product and framework versions independently where available.

Operations

Security guidance

  • Use a strong deployment-specific JWT secret.
  • Enable secure cookies behind HTTPS.
  • Restrict WebSocket allowed origins to trusted product origins.
  • Keep package tokens and registry credentials outside source control.
  • Replace all development Compose passwords.
  • Do not expose AI, MinIO, Redis, RabbitMQ, database or LiveKit secrets through frontend APIs.
  • Use backend @RequiresPermission checks even when a frontend action is hidden.
  • Review automatic inactive-user disablement before production rollout.
  • Audit role combinations and separation-of-duties warnings.
CSRF design note
Spring Security CSRF is disabled while authentication uses an HttpOnly SameSite=Lax cookie. Review this decision for each production deployment, particularly when cross-site embedding or non-idempotent browser requests are introduced.
Support

Troubleshooting

SymptomLikely checks
Login succeeds, then returns to loginCheck cookie domain/path, HTTPS secure-cookie setting, JWT secret consistency and /api/me response.
Route shows Access DeniedConfirm the route URL is present in /api/menu, the permission is active/granted and the active organization is correct.
Product menu is missingCheck LMNT_PRODUCT_CODE, enabled product record, menu permission and subscription access decision.
Chat says disconnectedCheck /chat-websocket, allowed origins, reverse-proxy WebSocket support, STOMP connection and Redis bridge configuration.
Incoming calls do not ringKeep the app tab open, verify the calls queue subscription, LiveKit credentials/URL and caller/callee communication scope.
AI is offlineCheck the Ollama/AI endpoint, selected model and /api/app/health.
Package install returns 401/403Verify classic PAT scopes, GitHub Packages login, .npmrc, Maven server ID and repository access.
Theme or language does not updateConfirm Settings permissions, theme API response, local theme mode and availability of the requested i18n bundle.
